# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Laravel 12 + Filament 5 hospitality management system covering three merged operational areas: **POS** (bar/restaurant ordering), **Hotel** (rooms/bookings/folios), and **Inventory/Accounting** (stock, shifts, staff debts). Built on the official Laravel/Livewire starter kit, now heavily domain-customized. Frontend is server-driven: Livewire/Volt + Filament, no SPA framework.

## Commands

```bash
composer setup        # first-time install: composer install, .env, key:generate, migrate, npm install+build
composer dev           # runs php artisan serve + queue:listen + vite dev, concurrently
composer lint          # pint --parallel (auto-fix)
composer test:lint     # pint --parallel --test (check only, no fix)
composer test          # config:clear + test:lint + php artisan test (full suite)

php artisan test --filter=testName          # single test by name
php artisan test tests/Feature/Path/File.php # single test file
./vendor/bin/pest tests/Feature/Path/File.php --filter="test description"
```

Tests run on **sqlite `:memory:`** (`phpunit.xml`), `RefreshDatabase` per test, queue `sync`, cache/session `array`. Real dev/prod DB is MySQL (`DB_CONNECTION` in `.env`).

**PHP version note:** `composer.json` requires `php: ^8.4`. If `php artisan`/`composer` commands fail with a Composer `platform_check.php` version error, the shell's default `php` binary is not the one this project needs — find the correct PHP 8.4 binary (e.g. a specific Laragon PHP version folder) rather than assuming the site is actually broken; the web server may already be configured with the correct version even when the CLI isn't.

**Deploy** (`deploy.sh`, production only): backs up DB → maintenance mode → git pull → composer install (no npm — compiled assets are committed to git) → migrate → `shield:generate`/seed roles **additively** (never removes/overwrites live-configured roles) → cache rebuild → queue restart → maintenance off.

## Architecture

### Two separate Filament panels

- **`admin`** (`/admin`, default) — the operational panel every staff role uses. `AdminPanelProvider` registers `FilamentShieldPlugin`.
- **`ceo`** (`/ceo`) — a read-only executive reporting panel, structurally isolated (own `discoverResources`/`discoverPages`/`discoverWidgets` roots under `app/Filament/Ceo/`) so operational admin pages can never leak into it. No Shield plugin here — see auth below.

Both are registered in `bootstrap/providers.php` and share the default `web` session guard, so a logged-in session carries over between them; `/ceo` is simply a URL a permitted user can navigate to directly (no cross-panel nav link exists).

### Dual authorization pattern — know which one applies before adding a new gate

This codebase gates things through **three** unrelated mechanisms depending on what you're adding:

1. **Filament Pages** (`app/Filament/Pages/*`, e.g. `WaiterLedger`, `StockValuation`) — gated via `PermissionService::canAccessPage(self::class)` in a `canAccess()` static method, backed by the `PagePermission` model and configured at runtime through the "Page Permissions Manager" admin UI. Deny-by-default except `super_admin`. No seed data — access is granted manually post-deploy.
2. **Filament Resources** (`app/Filament/Resources/*`, e.g. `ProductResource`) — gated through real Laravel Policies generated by Filament Shield (`php artisan shield:generate --resource=XResource --panel=admin`), producing `app/Policies/XPolicy.php`. **A new Resource with no generated policy is wide open to any authenticated panel user** — Filament does not deny by default the way PagePermission pages do. Grant permissions additively via `Role::givePermissionTo()`, never `syncPermissions()` on a live system.
3. **The `ceo` panel** — gated entirely by `User::canAccessPanel()` (`app/Models/User.php`): `super_admin` passes everywhere; the `ceo` role passes only on `panel-\>getId() === 'ceo'` and is explicitly blocked from every other panel; everything inside the panel is visible to whoever gets in (`app/Filament/Ceo/Concerns/CeoReadOnlyResource.php` forces `canViewAny()/canView()` true and all mutation checks false, since admin-panel Shield policies would otherwise silently apply to the same model classes and wrongly deny a ceo-only user).

### Two authentication guards — never let them cross

- `web` — normal Filament panel login (managers/admins/office staff), email+password.
- `staff_pin` — a **structurally separate** session guard for floor staff (`config/auth.php`), used by the kiosk/PIN system. Filament never looks at this guard, by design.

Flow: a device is trusted first (shared kiosk via a one-time registration code → `KioskDeviceService`, hashed token in an httpOnly `kiosk_device_token` cookie; or a personal device via `StaffLoginController` validating credentials *without* starting a `web` session, then `TrustedDeviceService::trust()`). Only once a device is trusted can staff authenticate with a 4-digit self-chosen PIN (`PinAuthService`) into the `staff_pin` guard. Middleware: `EnsureValidKioskDevice` / `EnsureTrustedDevice` / `EnsureStaffPinAuthenticated` (aliases in `bootstrap/app.php`), applied in `routes/web.php`. `PinAuthService::setPin` is self-service only; a manager can force-reset a PIN but never view/choose one.

### Domain map

- **POS**: `Order` is the hub — belongs to `Table`/`Booking` (dine-in vs room-charge), `Shift`, `User`/`Guest`; splits into bar/kitchen sub-orders via `OrderSplitter`, gated on an active bartender/chef `Shift`. `MenuItem` consumes `Ingredient`s via a `Recipe` pivot.
- **Hotel**: `Booking` → `Room` + `Guest`, with one `Folio` whose balance is always computed live from immutable `FolioLine`s (never stored). Room occupancy is likewise derived live from bookings, never stored on `Room`.
- **Inventory** — **two parallel, mirrored stock tracks by design, not an oversight**: `Product`/`InventoryItem`/`InventoryTransaction` (bar/POS-sold items) and `Ingredient`/`IngredientInventoryItem`/`IngredientTransaction` (kitchen-consumed items), each with its own transfer-item table. `WareHouse` is the shared pivot (bar/kitchen/main store) for both tracks plus `CountSession` handovers and `StockTransfer` routes. Four-eyes approval on `StockAdjustment` applies to everyone, including `super_admin` — no direct-apply exception.
- **Shift / Accounting**: `Shift` is one shared model/table across roles (`type`: waiter/cashier, bartender, chef, receptionist) but each role has its **own service** for divergent lifecycle rules — `ShiftAccountingService` (waiter/cashier expected totals from `OrderPayment`), `ReceptionistShiftService`, `BartenderChefShiftService` (handover-count *is* the shift boundary, no cash step), `CashierSettlementService` (shared channel-by-channel confirmation engine — cash/POS-machine/transfer — deliberately blind to the staff's own declared figures), `CashierSessionService` (separate concept: a cashier's own custody session, intentionally never gated on prior unclosed sessions). Shortfalls become `StaffDebt`.
- **Auth/Access**: `User` (Spatie `HasRoles`), `PagePermission`, `KioskDevice`/`KioskRegistrationCode`/`TrustedDevice`, `Setting`.
- **CEO reporting**: `DailyMetricsService` is the single shared calculation layer (dashboard, snapshot job, and Report Explorer all call into it); `DailyBusinessSnapshot` is an immutable daily snapshot computed by the scheduled `hms:compute-daily-snapshot` command.

### Business day boundary — 9am WAT, not midnight

`app/Support/BusinessDay.php` defines the trading day as closing at 9am `Africa/Lagos` (fixed UTC+1) rather than the more typical 4am, so a 1am sale belongs to "last night" and overnight staff still have until mid-morning to settle up, hand over, and reconcile before that business day closes out from under them. **Every date boundary in owner/CEO reporting must go through `BusinessDay`, never `Carbon::today()/yesterday()`** — otherwise the snapshot and the live dashboard disagree with each other for the same business day. Shifts themselves are unscheduled (no fixed handover clock); this boundary is purely a reporting convention layered on top.

### Enforced conventions (checked by an architecture test)

`tests/Feature/Architecture/NoBareAbortTest.php` fails the suite if any `abort()` call exists outside `app/Http/Middleware/`. Every other guard in the app (Livewire, Filament, services) must surface failures through `UserFeedback` or `Filament\Notifications\Notification` instead — a bare `abort()` drops the user on a blank framework error page with no cause/remedy message, which this system treats as a bug class, not a style preference.

### Before using a third-party package API

Grep the actual installed source under `vendor/<package>/src` before writing code against it — don't rely on memorized APIs from other projects/versions. This repo has been burned repeatedly by version drift in `spatie/laravel-activitylog` v5 (renamed namespaces/methods vs. common assumptions) and `spatie/laravel-permission` event payloads.
